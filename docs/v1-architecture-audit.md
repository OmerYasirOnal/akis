# PART A — Subsystem Maps

## Frontend chat / conversation screen — exhaustive subsystem map

**Summary.** The chat screen is the platform's centerpiece and its most overloaded surface. It is a four-level component stack: ChatPage (orchestrator, 510 lines) → ChatPageLayout (DOM shell, 457 lines, 57 props) → ChatPanel (576 lines, 60+ props) → {ChatMessage 1359-line dispatcher, PipelineDetailRail 946-line 4-tab hub, PreviewPanel 747-line Sandpack pane}. ChatPage already underwent an "F-06" decomposition into 16 custom hooks, but the orchestration didn't shrink — it relocated, and the page still carries 8 useState + 7 useRef + 8 useEffect + 22 distinct hook invocations and assembles a single 57-key prop bag that is drilled three layers deep. The screen renders at least a dozen independent feature surfaces (chat thread, plan card, 3 gates, failure cards, detail rail with Akış/Açıklama/Regresyon/AI-Logları tabs, cinema, explanation panel, preview panel, model picker, trace toggle, intent routing, onboarding). The core structural problem is a combinatorial state explosion: a 15-value ConversationUIState union and an 18-member ChatMessage discriminated union drive giant if/switch ladders that are duplicated across ChatPanel, ChatMessage, PipelineDetailRail, and the conversationToChatMessages mapper — so a single behavioral change (e.g. a new stage, or a gate) must be threaded through 4+ files, which is exactly the "fix one bug, spawn another" pain the user reports. The dual-render-path hazard (memory's known issue) is confirmed: plans render via two independent code paths (standalone 'plan' message vs embedded scribe_completed bubble) and the push gate renders in two physical places with split responsibilities.

### Recommendations

- **Replace the 57/60/45-prop pass-through chain with 2-3 React contexts (PipelineDataContext for workflow-derived fields, PipelineActionsContext for callbacks, ChatStreamContext for activities/SSE). Leaf components (PipelineDetailRail, PlanCard, gates) consume directly instead of being drilled through.** _(effort L, risk medium)_ — Eliminates the dominant prop-drilling thicket; adding a data field stops touching 4 files. This is the highest-leverage change for the user's 'fix one bug spawns another' pain and is low-risk because it is mechanical (move props into providers, no behavior change).
- **Extract ChatMessage's 18 cases into a registry of one-file-per-type renderers (messageRenderers[message.type]) and split the 285-line test_result case + inline helper components out of the monolith.** _(effort M, risk low)_ — The 1359-line switch is the single most change-hostile file; isolating each message type makes a per-type fix impossible to break adjacent cases, and lets 3 parallel sessions own different message types without colliding.
- **Collapse the two plan-render paths into one: always emit a single canonical plan ChatMessage from the mapper (prefer the scribe_completed embedded data when present, fall back to spec), so ChatMessage has exactly one plan render path.** _(effort M, risk medium)_ — Directly removes the confirmed dual-render hazard that memory flags as a recurring bug source; a plan-card change then lands in one place.
- **Centralize the ConversationUIState→behavior mapping in one module (e.g. uiStateMeta.ts exposing isRunning/isGate/agentInfo/glowColor/pollCadence per state) and have ChatPanel, PipelineDetailRail, the mapper, and ChatRouter import from it instead of re-listing states.** _(effort M, risk low)_ — Makes 'add a stage' a single-file edit and removes the four divergent state-ladders that currently must be kept in sync by hand.
- **Unify the push gate into one component that renders its announcement+actions together (in the rail, with an optional 'open preview' affordance), and have PreviewPanel embed that same component instead of a separate PushGateFooter.** _(effort M, risk medium)_ — Removes the split-responsibility hazard where the user can see the gate announcement but not the action buttons when the preview is closed; one source of truth for confirmPush/cancelPush.
- **Move PlanCard's Jira/Confluence fetching + status-branch logic into a dedicated IntegrationPickers component (or a hook) so PlanCard is presentation-only.** _(effort S, risk low)_ — Restores single responsibility to the most-visible gate card and lets integration UX evolve without risking the plan layout; reduces PlanCard from 555 to a presentational core.
- **For the 3-parallel-session goal: assign ownership boundaries along the seams above — Session A owns pages/chat/* + contexts (orchestration), Session B owns components/chat/ChatMessage + message-renderers (thread rendering), Session C owns components/pipeline/* (rail, cinema, gates, explanation). Land the context extraction (rec 1) first so the seams are clean before parallelizing.** _(effort S, risk low)_ — The current 57-prop coupling means any two sessions editing ChatPage/Layout/Panel will collide on the prop bag; introducing contexts creates stable interfaces that let the three sessions work the same screen without merge conflicts.


## Backend pipeline orchestrator (FSM, stage runners, iterate loops, SSE/explainability wiring, god-object smells)

**Summary.** PipelineOrchestrator.ts is a 2,877-line god class that owns the entire Scribe→Proto→Trace FSM, every lifecycle method (approve/reject/retry/cancel/confirmPush/cancelPush/skipTrace/iterate/criticOverride/toggleTrace), token accounting, GitHub auth, Jira/Confluence hooks, CI polling, chat-narrator duration tracking, and 9 injected sub-services. A real "Kademe 1/2/3" decomposition already happened — ~4,500 lines were pulled out into helpers/ (8 modules), stages/ (5 runners) and outcomes/ (2 discriminated-union handlers) — but the extraction was mechanical: the orchestrator now mostly re-wraps each extracted function inside a thin method that rebuilds a giant deps object (buildRunProtoAndTraceDeps, buildProtoTraceOutcomeDeps, buildTraceOutcomeDeps each hand-wire 15-25 closures), so the coupling didn't shrink, it just moved into deps-plumbing. The FSM has 16 stages but no single transition table — `store.update({stage})` is scattered across ~67 call sites in 9 files, and the legal-transition rules live implicitly inside per-method stage guards (`assertStage`, idempotency allowlists). Two big implicit-coupling hazards dominate: (1) `intermediateState` is an untyped `Record<string,unknown>` JSON bag carrying ~20 distinct keys (criticBlock, validationResult, traceIterateRetryCount, traceDryRunStatus, acCoverage, imageBlocks, …) that every stage read-modify-writes by hand, and (2) the orchestrator's constructor `emit` callback is wired to `undefined` in pipeline-factory, so the real SSE signal flows through a completely separate `pipelineBus`/`emitActivity` channel — `emitEvent` survives only for terminal cleanup + token flush, a dual-signal split that makes "did the frontend get the transition?" non-local. The Kademe-3 outcome unions genuinely fixed the forgotten-emit bug class for Proto/Trace, but Scribe and the iterate dispatchers were never converted and still emit inline.

### Recommendations

- **Introduce a typed IntermediateState interface and a single mergeIntermediateState(pipelineId, patch) helper that does the read-modify-write atomically (or via a DB-level JSONB merge), replacing the ~20 hand-rolled {...existing, key} spreads.** _(effort M, risk medium)_ — intermediateState is the #1 ripple/silent-bug source (a typo'd key no-ops; concurrent RMW loses keys). Typing it turns three-stages-away breakage into a compile error and centralizes the merge so the last-writer-wins race is fixable in one place. Directly attacks user pain #3 (one fix ripples).
- **Extract an explicit FSM module: a const transition table (from-stage → allowed to-stages, keyed by trigger) plus a single transition(pipelineId, to, {emit}) function that all 67 store.update({stage}) sites route through. Keep assertStage but derive its allowlists from the table.** _(effort L, risk medium)_ — There is currently no single place to see or enforce legal transitions; they're smeared across 9 files. A table makes the FSM auditable, lets a new parallel session reason about state safely, and is the natural home to also emit the SSE activity so transition-without-emit becomes structurally impossible (closing the remaining forgotten-emit gap on Scribe/lifecycle paths).
- **Unify the Trace and Critic iterate loops: one IterateLoop<T> abstraction parameterized by {counterKey, envCap, buildFeedback, shouldIterate}, replacing the twin evaluators in criticHelpers.ts and the twin dispatchers in iterateDispatchers.ts. Give the combined loop a single shared retry budget.** _(effort M, risk medium)_ — ~95% duplicated logic across 4 functions; two independent retry budgets feeding the same proto_building re-entry is a correctness smell (up to 6 unbounded-feeling loops). One abstraction halves the surface and makes the loop cap explicit and demo-safe.
- **Convert handleScribeResult and the inline-emitting lifecycle methods (rejectSpec, confirmPush, criticOverride) to the same ScribeOutcome / LifecycleOutcome discriminated-union + exhaustive-handler pattern already proven for Proto/Trace (Kademe 3).** _(effort M, risk low)_ — The forgotten-emit bug class is only half-eliminated — Scribe and the lifecycle methods still hand-sequence store.update+emit and are the remaining cinema/checkmark-bug risk. Reusing the existing proven pattern is low-novelty and finishes the job before defense.
- **Decide DevAgent + runIterationProtoAndTraceStage's relationship to the verification chain explicitly: either route them through a lightweight verify step (at least DeterministicValidator + optional Trace) or label them clearly as 'unverified dev mode' in the product. Do not leave a silent GitHub-push side door.** _(effort M, risk medium)_ — The platform's whole thesis is the verification chain; two paths (DevAgent push, iteration-mode completed) currently push code to GitHub with zero verification. For a graduation defense this is both a product-integrity gap and an easy committee question. (User has flagged DevAgent as 'döneceğiz' — this is the moment.)
- **For 3 parallel Claude sessions: assign ownership along the already-extracted seams — Session A owns helpers/ + outcomes/ (pure, deps-injected, low collision), Session B owns the FSM/transition-table + lifecycle methods in the main file, Session C owns the stage runners (runScribe/runProtoAndTrace/runTrace). The deps interfaces are the contract boundary; agree on them first and they can work without touching each other's bodies.** _(effort S, risk low)_ — The mechanical Kademe extraction, whatever its coupling cost, did create file-level seams that map cleanly to 3 non-overlapping work units. The main collision risk is the shared deps-builder methods and the PipelineStage union — freeze those interfaces up front. Directly serves the user's parallel-session requirement.
- **Move stageStartedAt / token accumulation duration tracking into the DB (or accept it as best-effort) and document that the orchestrator is single-instance; gate any future horizontal scaling on removing the in-memory Maps.** _(effort S, risk low)_ — In-RAM state (3 Maps) silently degrades on restart and diverges across processes — relevant now that parallel sessions may run separate backends on the same DB. Low urgency for the demo but worth a documented constraint so a parallel session doesn't assume durability.


## Bug archaeology — fragile hotspots, recurring patterns, and the coupling that makes fixes spawn regressions

**Summary.** Bugs cluster in two God-files and the thin "translation layers" that glue the backend pipeline state machine to the chat UI. The backend `PipelineOrchestrator.ts` (2877 LOC, 64 commit touches / 50 fix touches — the single highest-churn file by a 13x margin) and the frontend `ChatMessage.tsx` (1359 LOC) are the two God-files the user feels as "too crowded." But the real regression engine is the SSE/event translation ring: backend `activityEmitter.ts` → `PipelineOrchestrator.emitEvent` → `usePipelineStream.ts` → `PipelineCinema.utils.ts` → `workflows.ts mapConversation` → `conversationToChatMessages.ts`. The pipeline has no single source of truth for "what stage is the user looking at": stage truth is reconstructed three independent ways (SSE activity stream, polled pipeline snapshot, and FE-derived chat rows), so every fix that satisfies one path silently desyncs the others. Three structural couplings drive almost all regressions: (1) the orchestrator emits SSE events via 12 hand-duplicated `emitEvent` DI callbacks, so any new stage or branch can silently forget to emit; (2) the chat render layer reconstructs timestamps with `new Date()` at 18+ sites, so any event lacking a backend timestamp floats to "now"; (3) the plan card and Scribe bubble have a dual render path (legacy `spec_draft` vs new `scribe_completed`) that must be patched in two files at once. The verification chain is itself a recurring bug surface — five distinct iterate/retry/DevAgent code paths each re-implement (or skip) the Validator→Critic→Trace sequence instead of sharing one gated runner.

### Recommendations

- **Establish ONE authoritative stage model on the backend and make SSE + snapshot + cinema all read from it. Emit a single typed `PipelineStageView` (current stage, per-column status, last-event timestamp) that the SSE stream, the GET snapshot, and the cinema all consume verbatim — kill the 3 independent derivations.** _(effort L, risk medium)_ — The SSE-desync / cinema-empty cluster is the single most-repeated fix loop (5+ commits hitting the same 4-file ring). Every fix there is a patch on a symptom because no layer owns stage truth. A single emitted view object removes the disagreement that the patches keep chasing.
- **Extract one shared `postProtoQualityGates(ctx)` runner (Validator → Critic → Trace, with a dryRun flag) and route ALL five paths (normal, iteration, retryProto, executeIteration push-timing, DevAgent) through it. Make 'push to GitHub' physically unreachable except after the gate returns approved.** _(effort L, risk medium)_ — The verification chain is the platform's non-negotiable core, yet it is the most bypassed surface (P0-4, #683, DevAgent, iteration-verify-before-push all trace to re-implementations). A single gated runner makes bypass a compile-time impossibility instead of a per-path discipline. Directly serves the user's 'keep features but restructure cleanly' goal.
- **Delete the `new Date()` timestamp fallback everywhere and stamp every conversation event at write time on the backend; collapse mapConversation and conversationToChatMessages into ONE transcript builder.** _(effort M, risk low)_ — Two transcript builders with duplicated timestamp + dual-render logic is why #755/#758 only half-fixed the float bug (the carry-forward lives in the dead path). One builder + backend-stamped events makes render-time clocks impossible and removes a whole regression class.
- **Split ChatMessage.tsx into per-variant components (PlanBubble, FailureCard, AgentLine, UserMessage, GatePrompt) behind a small dispatcher, and give PlanCard a single render entry. Pair this with the chat-screen decluttering the user wants.** _(effort M, risk low)_ — ChatMessage.tsx (1359 LOC) is literally the 'chat screen too crowded' complaint; the dual PlanCard call sites are the dual-render bug. Per-variant components give the 3 parallel sessions non-overlapping files to edit, directly addressing the collision-avoidance requirement.
- **Centralize SSE emission: replace the 12 hand-duplicated emitEvent DI callbacks with one injected emitter object, and make every stage runner return a discriminated outcome (extend the Kademe-3 pattern to all lifecycle methods, not just runProtoAndTrace/runTrace).** _(effort L, risk medium)_ — Kademe-3 already proved this eliminates the forgotten-emit class for two runners; finishing the job removes it everywhere and shrinks the 2877-LOC orchestrator God-file the user feels as 'too crowded'.
- **Move intent classification at active gates off the regex fast-path: when pipeline state is awaiting_*, treat fix-command verbs (düzelt/güncelle/çek/ekle/değiştir) as FEEDBACK before any classifier runs, OR call the LLM classifier at gates.** _(effort S, risk low)_ — The gate intent-misroute is demo-critical (breaks the 'type to fix' verification-chain narrative) and is caused by the regex layer winning before the gate-aware fallback. Biasing at the gate is a small, contained change with high demo payoff.
- **For the 3 parallel sessions: add a migration-index reservation step (each session claims the next _journal.json index up front via the shared .claude/state coordination file) and enforce file-ownership lanes (BE-orchestrator / FE-chat-render / FE-cinema-SSE) so two sessions never edit the same hotspot ring.** _(effort S, risk low)_ — The orchestrator, ChatMessage, workflows.ts, and _journal.json are exactly the files most likely to collide; the existing coordination-protocol.md plus ownership lanes converts the worst-churn files into single-owner files per session. Matches the proven worktree pattern in MEMORY.


## Chat UI/UX complexity and usability degradation

**Summary.** The chat experience is a single deeply-nested render tree (ChatPanel → PipelineDetailRail → 4 tabs → cinema/explanation/regression/AI-logs, plus inline PushConfirmGate / CriticResolutionGate, plus a side PreviewPanel with its own push-gate footer, plus a 16-variant ChatMessage dispatcher) driven by a 12-member ConversationUIState union crossed with gate/failure/intent/collapse axes. ChatPanel takes ~40 props and ChatMessage is a 1359-line switch with 16 message types; PipelineDetailRail is a 947-line component that simultaneously hosts tabs, a mini stage-dot strip, Jira/Confluence/CI/attention pills, a resize handle, and two inline resolution gates. The combinatorial visual state space is in the thousands once gates × failure cards × intent modes × rail collapse are multiplied, and many of these surfaces fire at the same moment, competing for attention — the exact "her yer yazıya boğuk, kırmızı kırmızı uyarılarla çevrili" complaint already logged in the 2026-05-15 manual test. i18n is only half-adopted: the catalogue has 1273 TR/EN keys at parity, but the highest-traffic chat components leak large amounts of hardcoded Turkish (ChatInput has no useI18n import + 14 TR literals; PipelineDetailRail 22; ExplanationPanel 17; ChatMessage 9; PipelineCinema 9), so EN users see Turkish and every new string is a coin-flip on whether it goes through the catalogue. For the non-developer "bakkal" persona this is overwhelming: jargon (Spec/MVP/user story), three status chips in three different corners, duplicate stage-transition info in 3+ places, and gate affordances that don't match the chat input behavior. The core structural problem is that the chat panel has no single state model or surface budget — every feature was bolted on as another conditionally-rendered block keyed off uiState, so each new concern adds an independent affordance instead of fitting into a designed hierarchy.

# PART B — Strategy Panels

## Approach: Extract-behind-a-stable-interface, never rewrite-the-core. Every step is a mechanical "move/wrap, no behavior change" PR that ships independently, is reversible by revert, and keeps the live-AI demo path (Scribe to gate to Proto to Trace to push) byte-for-byte intact. We attack the two crowded God-files (PipelineOrchestrator.ts, the ChatPage to ChatMessage stack) by carving SEAMS that (a) make a single data/behavior change touch one file instead of four, and (b) give the 3 parallel sessions disjoint ownership lanes. We deliberately do NOT touch the verification-chain ordering in runProtoAndTrace.ts, do NOT convert all lifecycle methods at once, and do NOT redesign the FSM in one go. Backend and frontend always ship as separate PRs (per project convention). The first move is operational, not architectural: make parallel sessions physically unable to nuke each other and lock the verification chain with a guard test, so all subsequent refactor PRs land on a safe, regression-detecting base.

**Exec.** The audit is accurate and verified against the tree: PipelineOrchestrator.ts (2877 lines), ChatMessage.tsx (1359), the 57/60/45-prop drill chain, the dead emit callback (pipeline-factory.ts:512 `undefined, // emit`), the untyped `intermediateState?: Record<string,unknown>` (PipelineTypes.ts:434, 102 refs), the dual plan-render path (conversationToChatMessages.ts scribeCompletedExists + two PlanCard sites in ChatMessage.tsx:435/511), and the global-nuke dev-down.sh are all real. The root cause of all three user pains is identical: low-altitude monoliths with no vertical seams, where one enum (ConversationUIState / PipelineStage) and one untyped blob (intermediateState / the 57-prop bag) are read-modified by hand in 4+ files. The fix is NOT a rewrite — it is a sequence of 11 small extract-behind-interface PRs, each demo-safe and reversible, that turn those shared mutable surfaces into single-owner modules. Crucially, the codebase already has the proven templates: the Kademe-3 outcome-union with `_exhaustive: never` guards (outcomes/), the src/contexts/ pattern (AuthContext), and 13 extracted chat hooks. We reuse those, not invent. With 12 days to defense, the plan front-loads two zero-risk safety PRs (parallel-session isolation + a verification-chain guard test) so that every refactor afterward fails loudly instead of silently regressing the demo.

**Highest-leverage-first.** Do the safety pair FIRST, before any refactor: (1) a `dev-down.sh --this` cwd-scoped stop + parameterized `dev-up.sh --session N` (ports 3000+10N / 5173+10N, per-session DB akis_dev_sN), and (2) a single backend contract test that asserts the Proto to Validator to Critic to Trace to push-gate ordering in runProtoAndTrace.ts. Rationale: the user's #1 stated goal is "stop bug-fixes from spawning regressions," and the #1 reason that happens today is that there is NO automated tripwire — a refactor that breaks the verification chain or desyncs the chat is discovered by eye, at defense. The guard test converts "fix one bug, spawn another" into a red CI signal in seconds, on the most load-bearing invariant. The dev-script isolation removes the one catastrophic footgun (dev-down.sh globally pkills every tsx/vite across ALL worktrees and downs the shared DB — verified, no --this/--all scoping exists) that makes 3 parallel sessions dangerous. Both are tiny, reversible, touch nothing in the demo render/pipeline path, and every subsequent refactor PR depends on them existing. This is the move with the biggest payoff because it makes all 9 following PRs safe to attempt.
